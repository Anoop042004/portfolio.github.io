Business Use Case 2: Secure Content Delivery for OTT Platforms

Problem: OTT platforms like Netflix, Amazon Prime, and Disney+ face challenges in preventing unauthorized access, password sharing, and piracy.

Challenges: Detecting and blocking millions of unauthorized or suspicious sessions in real-time without affecting legitimate users.

Market Benefits:

  • Reduced unauthorized access
  • Improved user trust and compliance
  • Faster request validation at the edge

Design Techniques and Algorithms:

  • Hash Maps for Token Validation in Secure OTT Delivery

    OTT platforms use AWS CloudFront with Lambda@Edge to validate millions of JWT session tokens efficiently at the edge.

    Role of CloudFront:

    CloudFront delivers content globally and runs Lambda@Edge functions near users, enabling real-time token validation without hitting origin servers.

    How Hash Maps Work:<
    • JWT tokens include a kid (key ID) in their header.
    • Lambda@Edge caches kid â†’ RSA public key mappings in an in-memory hash map.
    • The kid is hashed for O(1) lookup of the public key.
    • On token arrival, the function extracts the kid, fetches the public key from the hash map, and verifies the JWT signature immediately.
    • If the kid isnâ€™t found, the function refreshes keys from Cognito and updates the cache.
    Benefits:
    • Constant-time key retrieval speeds up token validation.
    • Reduces backend load and latency.
    • Enables real-time blocking of unauthorized sessions at the edge.

    This hashing-based caching system ensures secure, scalable, and fast validation for OTT platforms.

  • Bloom Filters for Space-Efficient Blacklist Checks

    OTT platforms often block known pirated session tokens or device IDs using large blacklists. But storing these lists directly at the edge can consume significant memory and increase lookup time.

    Role of Bloom Filters:

    Bloom Filters allow space-efficient representation of blacklists by using bit arrays and multiple hash functions. They support constant-time membership checks with acceptable false positives but no false negatives.

    How Bloom Filters Work:
    • Each blacklisted token/device ID is hashed using multiple hash functions.
    • Each hash function sets a bit in a fixed-size bit array.
    • To check if a token is blacklisted, it is hashed again using the same functions.
    • If all corresponding bits are set, the token is probably in the blacklist.
    • If any bit is not set, it is definitely not in the blacklist.
    Benefits:
    • Efficient O(k) time complexity, where k is number of hash functions (usually small).
    • Much lower space requirement than storing entire lists.
    • Ideal for deployment at edge locations with limited memory.
    • Helps block suspicious tokens/devices without frequent server lookups.

    Bloom Filter C++ Code

    #include <iostream>
#include <bitset>
#include <vector>
#include <string>
#include <functional>

const int BLOOM_SIZE = 1000;
const int HASH_COUNT = 3;

class BloomFilter {
  std::bitset<BLOOM_SIZE> bitArray;

  size_t hash(const std::string& data, int seed) const {
    std::hash<std::string> hasher;
    return hasher(data + std::to_string(seed)) % BLOOM_SIZE;
  }

public:
  void insert(const std::string& item) {
    for (int i = 0; i < HASH_COUNT; i++) {
      bitArray.set(hash(item, i));
    }
  }

  bool possiblyContains(const std::string& item) const {
    for (int i = 0; i < HASH_COUNT; i++) {
      if (!bitArray.test(hash(item, i)))
        return false;
    }
    return true;
  }
};

int main() {
  BloomFilter filter;
  filter.insert("token123");
  filter.insert("piratedUser456");
  filter.insert("device789");

  std::cout << "Check token123: " << filter.possiblyContains("token123") << "\n";
  std::cout << "Check userX: " << filter.possiblyContains("userX") << "\n";
}

Business Use Case 3: Efficient Routing for Content Delivery

Problem Statement:

  • Traditional CDN routing mechanisms rely on static or latency-only decisions.
  • This can lead to inefficient delivery paths under changing network conditions.
  • Results in delays and increased packet loss.

Solution Approach:

  • Integrate a custom intelligent routing microservice.
  • Use the A* algorithm to dynamically select the most optimal edge server or path.
  • Leverage AWS CloudFront with Lambda@Edge or CloudFront Functions as the entry point.

Data Structures & Algorithms:

  • Graph Data Structure: Represents CDN edge locations and connections with weights like latency or hops.
  • A* Algorithm:
    • Uses heuristics to find the optimal path through the network graph.
    • Efficiently chooses the best edge location or transit route in real-time.
    • Time Complexity: O(E), where E is the number of edges (efficient for sparse graphs).

How It Works:

  • Client requests content from CloudFront.
  • A Lambda@Edge function is triggered.
  • This function sends live metrics (latency, packet loss, etc.) to the routing microservice.
  • The microservice builds a weighted graph of possible paths.
  • The A* algorithm calculates the optimal delivery path.
  • CloudFront uses the result to route the content efficiently.

A* Algorithm Implementation:

  • Dijkstraâ€™s Algorithm

Use Case Fit:
Best suited for content routing when all edge weights (such as network latencies) are non-negative. Helps select the shortest and most efficient path from the origin (e.g., CloudFront edge) to multiple destination nodes.

Working:
The algorithm begins at the source node and explores neighboring nodes with the smallest cumulative latency. It uses a min-heap (priority queue) to always expand the nearest unexplored node. This process continues until the shortest paths to all nodes are identified.

Time Complexity:
O((V + E) log V) â€” efficient for large sparse graphs where:
V = number of nodes (edge locations)
E = number of connections (network paths)

Dijkstraâ€™s Algorithm Implementation:

Dijkstraâ€™s C++ Code


#include <iostream>
#include <vector>
#include <queue>
#include <unordered_map>
#include <limits>
using namespace std;

typedef pair<int, string> pii;

unordered_map<string, int> dijkstra(
    unordered_map<string, vector<pair<string, int>>> &graph,
    string start
) {
    unordered_map<string, int> distances;
    for (auto &node : graph) {
        distances[node.first] = INT_MAX;
    }
    distances[start] = 0;

    priority_queue<pii, vector<pii>, greater<pii>> pq;
    pq.push({0, start});

    while (!pq.empty()) {
        auto [currDist, currNode] = pq.top();
        pq.pop();

        if (currDist > distances[currNode])
            continue;

        for (auto &[neighbor, weight] : graph[currNode]) {
            int distance = currDist + weight;
            if (distance < distances[neighbor]) {
                distances[neighbor] = distance;
                pq.push({distance, neighbor});
            }
        }
    }

    return distances;
}

int main() {
    unordered_map<string, vector<pair<string, int>>> graph;
    graph["A"] = {{"B", 2}, {"C", 5}};
    graph["B"] = {{"A", 2}, {"C", 1}, {"D", 3}};
    graph["C"] = {{"A", 5}, {"B", 1}, {"D", 2}};
    graph["D"] = {{"B", 3}, {"C", 2}};

    string start = "A";
    auto result = dijkstra(graph, start);

    cout << "Shortest distances from node " << start << ":\n";
    for (auto &[node, dist] : result) {
        cout << "  " << node << " : " << dist << "\n";
    }

    return 0;
}

Limitation:
Does not support graphs with negative edge weights, making it unsuitable for routing decisions based on metrics like penalized or degraded paths.

Data Structures and Algorithms

Fenwick Tree (Binary Indexed Tree)

  • Purpose: Efficient support for score updates and prefix sum queries
  • Time Complexities:
    • update(index, value) â†’ O(log n)
    • query(index) â†’ O(log n)
  • Why Used: In a leaderboard with millions of players, Fenwick Tree scales efficiently where naive methods would not.

Fenwick Tree - Python Pseudocode

# #include 
#include 
using namespace std;

class FenwickTree {
private:
    vector BIT; // Binary Indexed Tree array
    int n;

public:
    // Constructor: initialize tree with size n
    FenwickTree(int size) {
        n = size;
        BIT.resize(n + 1, 0); // 1-based indexing
    }

    // Update function: add 'val' to index 'i'
    void update(int i, int val) {
        while (i <= n) {
            BIT[i] += val;
            i += i & (-i); // Move to parent
        }
    }

    // Get sum from 1 to i
    int getSum(int i) {
        int sum = 0;
        while (i > 0) {
            sum += BIT[i];
            i -= i & (-i); // Move to child
        }
        return sum;
    }

    // Optional: Get sum from index l to r
    int rangeSum(int l, int r) {
        return getSum(r) - getSum(l - 1);
    }
};

// Sample usage
int main() {
    vector arr = {0, 2, 1, 1, 3, 2, 3}; // 1-based, arr[0] is dummy
    int n = arr.size() - 1;

    FenwickTree ft(n);

    // Build the BIT
    for (int i = 1; i <= n; i++) {
        ft.update(i, arr[i]);
    }

    cout << "Prefix sum up to index 5: " << ft.getSum(5) << endl;
    cout << "Sum from index 2 to 5: " << ft.rangeSum(2, 5) << endl;

    // Update index 3 by adding 2
    ft.update(3, 2);
    cout << "After update, prefix sum up to index 5: " << ft.getSum(5) << endl;

    return 0;
}
Example Workflow

  • To display the playerâ€™s rank, the system calls query(index - 1) to determine how many players have higher scores.
  • Top N players can be retrieved using repeated queries or via a segment tree if required.

Data Structures and Algorithms

Trie

  • Used for instant search auto-suggestions as users type product names.
  • Time complexity: O(N*L) where N is the number of words in the trie and L is the length of the longest word in the trie.
  • The trie holds a set of product names or past search queries.
  • As a user begins typing (e.g., "hel"), the trie is traversed character by character.
    • Once a node representing the prefix is reached:
  • If it's a complete word (marked by isWordEnd), it is returned as one suggestion.
  • All children of that node are recursively searched to collect further suggestions.
  • If the prefix doesnâ€™t exist, no matches are returned.

Trie Implementation - C++

// C++ program to demonstrate auto-complete feature
// using Trie data structure.
#include 
using namespace std;

// Alphabet size (# of symbols)
#define ALPHABET_SIZE (26)

// Converts key current character into index
// use only 'a' through 'z' and lower case
#define CHAR_TO_INDEX(c) ((int)c - (int)'a')

// trie node
struct TrieNode {
    struct TrieNode* children[ALPHABET_SIZE];

    // isWordEnd is true if the node represents
    // end of a word
    bool isWordEnd;
};

// Returns new trie node (initialized to NULLs)
struct TrieNode* getNode(void)
{
    struct TrieNode* pNode = new TrieNode;
    pNode->isWordEnd = false;

    for (int i = 0; i < ALPHABET_SIZE; i++)
        pNode->children[i] = NULL;

    return pNode;
}

// If not present, inserts key into trie.  If the
// key is prefix of trie node, just marks leaf node
void insert(struct TrieNode* root, const string key)
{
    struct TrieNode* pCrawl = root;

    for (int level = 0; level < key.length(); level++) {
        int index = CHAR_TO_INDEX(key[level]);
        if (!pCrawl->children[index])
            pCrawl->children[index] = getNode();

        pCrawl = pCrawl->children[index];
    }

    // mark last node as leaf
    pCrawl->isWordEnd = true;
}

// Returns 0 if current node has a child
// If all children are NULL, return 1.
bool isLastNode(struct TrieNode* root)
{
    for (int i = 0; i < ALPHABET_SIZE; i++)
        if (root->children[i])
            return 0;
    return 1;
}

// Recursive function to print auto-suggestions for given
// node.
void suggestionsRec(struct TrieNode* root,
                    string currPrefix)
{
    // found a string in Trie with the given prefix
    if (root->isWordEnd)
        cout << currPrefix << endl;

    for (int i = 0; i < ALPHABET_SIZE; i++)
        if (root->children[i]) {
            // child node character value
            char child = 'a' + i;
            suggestionsRec(root->children[i],
                           currPrefix + child);
        }
}

// print suggestions for given query prefix.
int printAutoSuggestions(TrieNode* root, const string query)
{
    struct TrieNode* pCrawl = root;
    for (char c : query) {
        int ind = CHAR_TO_INDEX(c);

        // no string in the Trie has this prefix
        if (!pCrawl->children[ind])
            return 0;

        pCrawl = pCrawl->children[ind];
    }
    // If prefix is present as a word, but
    // there is no subtree below the last
    // matching node.
    if (isLastNode(pCrawl)) {
        cout << query << endl;
        return -1;
    }
    suggestionsRec(pCrawl, query);
    return 1;
}

// Driver Code
int main()
{
    struct TrieNode* root = getNode();
    insert(root, "hello");
    insert(root, "dog");
    insert(root, "hell");
    insert(root, "cat");
    insert(root, "a");
    insert(root, "hel");
    insert(root, "help");
    insert(root, "helps");
    insert(root, "helping");
    int comp = printAutoSuggestions(root, "hel");

    if (comp == -1)
        cout << "No other strings found with this prefix\n";

    else if (comp == 0)
        cout << "No string found with this prefix\n";

    return 0;
}

Segment Tree

  • Used to handle range queries like filtering products by price.
  • Time complexity: O(log n) for updates and queries.
  • The catalog prices are stored in an array, which is converted into a segment tree.
  • Each node of the tree represents a segment (sub-array) and stores useful information such as min price, max price, or product count in that range.
  • When a user applies a price filter (e.g., â‚¹5000 to â‚¹8000), the segment tree is queried for matching products in O(logâ€¯n) time.
  • This avoids scanning all products, reducing server load and improving speed.

Segment Tree Implementation - C++

#include 
#include 
using namespace std;

class SegmentTree {
private:
    vector tree;
    int n;

    void build(vector& prices, int node, int start, int end) {
        if (start == end) {
            tree[node] = prices[start];
        } else {
            int mid = (start + end) / 2;
            build(prices, 2 * node, start, mid);
            build(prices, 2 * node + 1, mid + 1, end);
            tree[node] = min(tree[2 * node], tree[2 * node + 1]); // store min price
        }
    }

    int rangeMinQuery(int node, int start, int end, int l, int r) {
        if (r < start || end < l) return INT_MAX;  // No overlap
        if (l <= start && end <= r) return tree[node]; // Total overlap

        int mid = (start + end) / 2;
        int left = rangeMinQuery(2 * node, start, mid, l, r);
        int right = rangeMinQuery(2 * node + 1, mid + 1, end, l, r);
        return min(left, right);
    }

public:
    SegmentTree(vector& prices) {
        n = prices.size();
        tree.resize(4 * n);
        build(prices, 1, 0, n - 1);
    }

    int query(int l, int r) {
        return rangeMinQuery(1, 0, n - 1, l, r);
    }
};

int main() {
    vector prices = {3000, 8000, 12000, 15000, 7000};
    SegmentTree st(prices);

    cout << "Minimum price from index 1 to 4: â‚¹" << st.query(1, 4) << endl;
    return 0;
}

Business Use Case 6: API Gateway Acceleration & DDoS Protection

Problem Statement: APIs serving mobile apps, IoT, and web platforms face slowdowns due to high request volumes and repeated calls. Backend servers can get overloaded, and APIs are vulnerable to attacks like DDoS, SQL injection, and bot scraping, risking downtime and data breaches.

Theory:

  • API responses must be fast and reliable even during traffic spikes.
  • Security layers like Web Application Firewalls (WAF) and DDoS protection are essential.
  • Using caching at edge locations reduces backend load.
  • Combining AWS CloudFront with AWS WAF and AWS Shield protects APIs and accelerates delivery.

Real-Time Usage:

  • CloudFront caches frequent API GET requests at edge locations worldwide for instant response.
  • AWS WAF filters malicious requests in real time to block threats like SQLi and XSS.
  • AWS Shield Standard automatically mitigates common DDoS attacks.
  • This combination ensures APIs remain fast, scalable, and secure during heavy traffic or attacks.

Data Structures and Algorithms:

  • Caching Mechanism: Stores frequently requested API responses at edge nodes to reduce server hits.
  • Rule-based Filtering: AWS WAF applies predefined or custom security rules to filter malicious traffic efficiently.
  • Traffic Absorption: CloudFrontâ€™s distributed network absorbs large traffic volumes, helping prevent origin overload.

Example Workflow:

  • A mobile app sends frequent API requests (e.g., fetching user data).
  • CloudFront edge location serves cached GET responses instantly.
  • AWS WAF inspects incoming requests and blocks suspicious patterns.
  • AWS Shield detects and mitigates volumetric DDoS attacks automatically.
  • Backend servers handle only legitimate, uncached requests, maintaining performance.
